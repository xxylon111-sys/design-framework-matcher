# 设计框架匹配器 (Design Framework Matcher)

根据用户的设计需求，通过结构化的三步流程匹配最合适的设计框架，确保后续设计生成和修改在统一框架约束下进行。

## 执行流程

严格按以下三步顺序执行，每一步必须获得用户确认后才能进入下一步。

---

### Step 1：项目判断

根据用户输入的设计需求内容，判断该需求属于哪个项目。

**当前支持的项目列表：**

| 项目代号 | 项目名称 | 说明 |
|---------|---------|------|
| BJH | 百家号 | 百度内容创作者平台，包含创作管理、数据分析、变现等功能模块 |
| TM | 天牧 | 天牧平台相关设计 |
| DJPC | 度加PC | 度加PC端相关设计 |

**执行规则：**
- 根据需求中的关键词、功能描述、业务场景自动判断最可能的项目
- 使用 AskUserQuestion 工具向用户展示判断结果，让用户**单选确认**
- 如果无法判断，列出所有项目让用户选择
- 用户确认后锁定项目，进入 Step 2

**判断依据示例：**
- 提到"创作者""发布""作品管理""收入""百家号" → BJH
- 提到"天牧" → TM
- 提到"度加""剪辑""PC端工具" → DJPC

---

### Step 2：需求语义拆解

确认项目后，对用户的设计需求进行语义拆解分析。

**拆解维度：**

1. **场景匹配**：判断需求属于项目框架文档中的哪个场景（参照 `references/frameworks/` 目录下对应项目的框架文档）
2. **内容拆解**：识别需求中包含的具体设计内容/模块
3. **重合度分析**：对比框架文档中已有模块，标注哪些是已有模块（可复用），哪些是新增模块

**输出格式（向用户展示）：**

```
📋 需求语义拆解结果

🎯 匹配场景：[场景名称]
   场景说明：[来自框架文档的场景描述]

📦 设计内容拆解：
   ✅ [已有模块1] — 与框架文档中 [模块名] 重合，可直接复用
   ✅ [已有模块2] — 与框架文档中 [模块名] 重合，可直接复用
   🆕 [新增模块1] — 框架文档中无对应，需新增设计
   🆕 [新增模块2] — 框架文档中无对应，需新增设计

📊 复用率：X/Y 个模块可复用（XX%）
```

展示给用户确认后进入 Step 3。

---

### Step 3：框架匹配与选择

根据 Step 2 的拆解结果，从框架文档中匹配最适合的框架。

**匹配规则：**
- 基于需求内容与各框架的场景适配度进行打分
- **必须提供 2-4 个候选框架**，不能只给一个
- 每个候选需包含：框架编号、名称、适配度说明、框架结构示意
- 按匹配度从高到低排列，第一个为推荐选项

**展示格式（每个候选框架）：**

```
┌─────────────────────────────────────────┐
│ [F编号] [框架名称]          匹配度: ★★★★☆ │
├─────────────────────────────────────────┤
│                                         │
│  ┌──────┬──────────────────┬────────┐   │
│  │      │                  │        │   │
│  │ 左   │    主内容区       │  右侧  │   │
│  │ 导   │  [布局描述]      │  边栏  │   │
│  │ 航   │                  │        │   │
│  │      │                  │        │   │
│  └──────┴──────────────────┴────────┘   │
│                                         │
│ 适配理由：[为什么这个框架适合该需求]       │
│ 包含模块：[该框架的核心模块列表]          │
└─────────────────────────────────────────┘
```

**使用 AskUserQuestion 工具让用户单选确认。**

**锁定规则：**
- 用户选择确认后，该框架即被**锁定**
- 将锁定的框架信息输出为明确声明
- 后续所有设计生成和修改操作必须在该框架结构约束下进行
- 不允许脱离锁定框架的布局结构

**锁定确认输出：**

```
🔒 框架已锁定

项目：[项目名称]
场景：[场景名称]
框架：[F编号] [框架名称]

布局结构：
[框架的完整布局描述]

组成模块：
- [模块1]：[说明]
- [模块2]：[说明]
- ...

⚠️ 后续设计生成和修改将严格遵循此框架结构。
如需更换框架，请重新执行 /design-framework-matcher。
```

## 注意事项

1. **严格按顺序执行**：Step 1 → Step 2 → Step 3，不能跳步
2. **每步需确认**：每一步都必须获得用户明确确认才能继续
3. **单选约束**：Step 1 项目选择和 Step 3 框架选择都只允许单选
4. **锁定不可逆**：框架一旦锁定，当次设计任务内不可更改（除非用户重新触发此 skill）
5. **框架约束传递**：锁定的框架信息需传递给后续的设计生成流程（如 pencil-design-workflow）
6. **文档优先**：所有框架定义以框架文档为准，不要凭空编造框架内容
