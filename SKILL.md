# 设计框架匹配器 (Design Framework Matcher)

根据用户的设计需求，通过结构化流程匹配最合适的设计框架，并在框架锁定后补充基于语义的组件识别和组装，确保后续设计生成和修改同时遵循统一框架约束与现有组件复用约束。

## 执行流程

严格按以下顺序执行，并在进入后续设计生成前产出明确的项目、框架锁定结果和组件语义复用结果。框架匹配必须先于组件选择；组件只能在锁定框架内表达内容层和交互层，不得反向改变框架结构。

### 交互确认适配规则

- 优先使用可用的交互式提问工具（如 AskUserQuestion、request_user_input）完成 Step 1 项目确认和 Step 3 框架选择。
- 如果当前运行环境没有交互式提问工具，但用户需求已经足够明确，可根据匹配置信度继续执行：
  - 高置信度：直接说明“自动确认/自动锁定”的项目、场景、框架和原因，然后继续后续设计任务。
  - 中低置信度：用一条简短问题向用户确认，不继续生成或修改设计。
- 用户在请求中明确点名项目、框架、页面来源或要求“直接生成/继续”，视为对对应步骤的确认；仍需在输出中写明锁定结果。
- 自动锁定必须可被用户纠正：如果用户随后指定其他项目或框架，重新执行本 skill 并覆盖本次锁定。

### 框架匹配口径

- 框架只定义稳定的页面区域关系、主次关系、控制关系和辅助关系；不要把表格、卡片、图表、按钮、输入框、空状态等组件形态当作框架类型。
- 功能块属于内容层或组件层，可用于判断场景和复用资源，但不能替代框架结构判断。
- 百家号 (BJH) 框架以 GitHub v2 资料为准：5 种区域关系框架，16 张参考图；需要视觉参照时读取 `references/frameworks/bjh-framework-visual.html` 和 `references/frameworks/百家号框架参考图/`。

### 组件语义匹配口径

- 组件不是 UI 元素清单，而是功能语义承载器。不要把“按钮”直接匹配成 Button，把“表格”直接匹配成 Tabular；先判断功能意图，再选择组件模式。
- 对百家号 (BJH) 页面，在 Step 3 框架锁定后必须读取 `references/components/bjh-component-semantics.md`，执行 Step 4 组件语义识别与组装。
- 组件识别链路固定为：PRD 自然语言 → 功能语义 → 用户任务 → 交互意图 → 组件候选 → 组件组合 → 设计结构。
- 若百家号组件语义库已有组件可表达需求，必须复用已有组件和组合；只有无法覆盖时才标注“缺口组件/待补充”，不得直接自创样式或组件。

### Step 1：项目判断

根据用户输入的设计需求内容，判断该需求属于哪个项目。

**当前支持的项目列表：**

| 项目代号 | 项目名称 | 说明 |
|---------|---------|------|
| BJH | 百家号 | 百度内容创作者平台，包含创作管理、数据分析、变现等功能模块 |
| TM | 天牧 | 天牧平台相关设计 |
| DJPC | 度加PC | 度加PC端相关设计 |

**执行规则：**
- 根据需求中的关键词、功能描述、业务场景自动判断最可能的项目
- 优先使用 AskUserQuestion/request_user_input 向用户展示判断结果，让用户**单选确认**；如果工具不可用，按“交互确认适配规则”处理
- 如果无法判断，列出所有项目让用户选择
- 用户确认后锁定项目，进入 Step 2

**判断依据示例：**
- 提到"创作者""发布""作品管理""收入""百家号" → BJH
- 提到"天牧" → TM
- 提到"度加""剪辑""PC端工具" → DJPC

---

### Step 2：需求语义拆解

确认项目后，对用户的设计需求进行语义拆解分析。

**拆解维度：**

1. **场景匹配**：判断需求属于项目框架文档中的哪个场景（参照 `references/frameworks/` 目录下对应项目的框架文档）
2. **区域关系拆解**：识别页面的主工作区、辅助栏、对象集合、详情处理区、叙事流、局部控制区等稳定结构关系
3. **内容层拆解**：识别需求中包含的具体功能块/组件（如表格、筛选、余额卡、提示条、按钮等），仅作为场景和复用判断依据
4. **重合度分析**：对比框架文档中已有场景和区域关系，标注哪些关系可直接复用，哪些内容层模块需新增设计

**输出格式（向用户展示）：**

```
📋 需求语义拆解结果

🎯 匹配场景：[场景名称]
   场景说明：[来自框架文档的场景描述]

📦 设计内容拆解：
   ✅ [区域关系1] — 与框架文档中 [场景/关系] 重合，可直接复用
   ✅ [区域关系2] — 与框架文档中 [场景/关系] 重合，可直接复用
   🆕 [内容层模块1] — 框架文档中无对应内容，可在锁定框架内新增设计
   🆕 [内容层模块2] — 框架文档中无对应内容，可在锁定框架内新增设计

📊 复用率：X/Y 个区域关系或场景模式可复用（XX%）
```

展示给用户确认后进入 Step 3；如果用户已要求直接生成且匹配置信度高，可展示拆解结果并继续。

---

### Step 3：框架匹配与选择

根据 Step 2 的拆解结果，从框架文档中匹配最适合的框架。

**匹配规则：**
- 基于需求内容与各框架的场景适配度进行打分
- **必须提供 2-4 个候选框架**，不能只给一个
- 每个候选需包含：框架编号、名称、适配度说明、框架结构示意
- 按匹配度从高到低排列，第一个为推荐选项
- 对百家号需求，优先在 5 种 GitHub v2 框架中候选：F1 双列聚合、F2 单工作区、F3 主从处理、F4 发现浏览、F5 叙事转化

**展示格式（每个候选框架）：**

```
┌─────────────────────────────────────────┐
│ [F编号] [框架名称]          匹配度: ★★★★☆ │
├─────────────────────────────────────────┤
│                                         │
│  ┌──────┬──────────────────┬────────┐   │
│  │      │                  │        │   │
│  │ 左   │    主内容区       │  右侧  │   │
│  │ 导   │  [布局描述]      │  边栏  │   │
│  │ 航   │                  │        │   │
│  │      │                  │        │   │
│  └──────┴──────────────────┴────────┘   │
│                                         │
│ 适配理由：[为什么这个框架适合该需求]       │
│ 包含模块：[该框架的核心模块列表]          │
└─────────────────────────────────────────┘
```

**优先使用 AskUserQuestion/request_user_input 工具让用户单选确认；如果工具不可用，按“交互确认适配规则”处理。**

**锁定规则：**
- 用户选择确认后，该框架即被**锁定**
- 自动确认场景下，第一个高匹配候选可被锁定，但必须说明自动锁定原因
- 将锁定的框架信息输出为明确声明
- 后续所有设计生成和修改操作必须在该框架结构约束下进行
- 不允许脱离锁定框架的布局结构

**锁定确认输出：**

```
🔒 框架已锁定

项目：[项目名称]
场景：[场景名称]
框架：[F编号] [框架名称]
确认方式：[用户选择 / 自动锁定]
自动锁定原因：[仅自动锁定时填写；用户选择时省略]

布局结构：
[框架的完整布局描述]

组成模块：
- [模块1]：[说明]
- [模块2]：[说明]
- ...

⚠️ 后续设计生成和修改将严格遵循此框架结构。
如需更换框架，请重新执行 /design-framework-matcher。
```

### Step 4：组件语义识别与组装（BJH 必执行）

框架锁定后，对百家号需求必须继续执行组件语义识别。其他项目若尚无组件语义库，说明组件语义库暂未建立，只输出框架约束和需补充的组件复用清单。

**执行规则：**
- 读取 `references/components/bjh-component-semantics.md`。
- 将需求自然语言拆成“功能语义、用户任务、交互意图”，再映射到现有组件候选。
- 以百家号 PC 组件库中的真实组件语义为准，优先复用 Button、Input、Select、DatePicker、Tabs、Tabular、Pagination、DataDisplay、Upload、Empty、Alert、Modal、Toast、Notification、Loading、Tag、Badge、Tooltip、Popover 等已有组件。
- 输出组件组合，而不是孤立组件清单。例如输出 `FilterBar + Tabular + Pagination`，不要只输出 `Input、Select、Table`。
- 对每个组件/组合写明适用语义、不适用边界、复用依据和是否存在缺口。
- 若后续要在 Figma、Pencil 或代码中生成 UI，必须按 Step 4 的组件组合搭建，不得随意新增未在语义库中登记的组件形态。

**输出格式：**

```
🧩 组件语义识别结果

功能语义：
- [语义1]：[来自需求的自然语言证据]
- [语义2]：[来自需求的自然语言证据]

组件候选：
- [组件]：适用语义 / 不适用边界 / 复用依据
- [组件]：适用语义 / 不适用边界 / 复用依据

组件组合：
- [组合1]：承载 [用户任务/交互意图]
- [组合2]：承载 [用户任务/交互意图]

复用校验：
- 已复用：[组件/组合]
- 不新增：[说明哪些需求由已有组件承载]
- 缺口：[仅当确实无法复用时填写，否则写“无”]
```

**常见意图映射示例：**
- 对象集合管理 → Tabs + FilterBar + Tabular + Pagination + Empty
- 查询筛选 → Input + Select + DatePicker/SelectDate + Button
- 批量操作 → Tabular.rowSelection + BatchActionBar + Button + Toast/Modal
- 数据分析 → Tabs + SelectDate/DatePicker + DataDisplay + Tooltip + Progress
- 表单配置 → FormItem + Input/Textarea/Select/Radio/Checkbox/Switch + ButtonGroup
- 上传提交 → Upload + Alert + Progress/Loading + Button
- 风险确认 → Button.danger + Modal + ButtonGroup + Toast
- 空数据引导 → Empty + Button/Link

## 注意事项

1. **严格按顺序执行**：Step 1 → Step 2 → Step 3 → Step 4，不能跳步；非 BJH 项目可在 Step 4 说明组件语义库缺失
2. **确认需适配环境**：有交互式提问工具时必须单选确认；无工具且高置信度时可自动确认并清楚声明假设
3. **单选约束**：Step 1 项目选择和 Step 3 框架选择都只允许单选
4. **锁定不可逆**：框架一旦锁定，当次设计任务内不可更改（除非用户重新触发此 skill）
5. **框架约束传递**：锁定的框架信息需传递给后续的设计生成流程（如 pencil-design-workflow）
6. **文档优先**：所有框架定义以框架文档为准，不要凭空编造框架内容
7. **不中断明确任务**：当用户已经提供完整页面需求并要求生成/修改 UI 时，不要因缺少交互式提问工具而停在推荐阶段；可自动锁定后继续执行
8. **组件不等于框架**：表格、卡片、图表、列表、按钮等只作为内容层和复用线索，不作为框架分类依据
9. **组件复用优先**：百家号页面必须先查组件语义库再搭建界面；不要自创视觉样式、组件名或组件组合
